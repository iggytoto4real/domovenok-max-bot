# DTO в backend-core: иммутабельность и контракт

## Принцип

Все DTO в проекте **иммутабельные**: создаются один раз со всеми данными, без сеттеров и последующего изменения состояния.

Зачем:

- **Предсказуемость** — объект не может «донаполниться» или измениться после создания; нет риска забыть выставить поле или перезаписать его по ошибке.
- **Поток данных явный** — кто создаёт DTO, тот и обязан передать все поля (профиль + баланс, ошибка и т.д.).
- **Проще тестировать и рассуждать** — нет скрытых мутаций.

## Реализация

- Используем **Lombok `@Value`**: поля считаются `final`, генерируются только геттеры, один полный конструктор. Сеттеров нет.
- **Создание** — через конструктор или статический фабричный метод с полным набором аргументов. Не создаём DTO «пустым» и не заполняем сеттерами потом.
- **Request DTO** (тело запроса, десериализация из JSON): один фабричный метод с `@JsonCreator` и `@JsonProperty` на параметрах, чтобы Jackson собирал объект в один вызов без мутаций.

## Примеры

**Ответ (только сериализуется в JSON):** создаётся в сервисе/контроллере со всеми полями.

```java
@Value
public class AuthInitResponse {
    UserDto user;
    String token;
    boolean firstVisit;
}
// Использование: new AuthInitResponse(user, token, firstVisit);
```

**Запрос (десериализуется из JSON):** фабричный метод для Jackson.

```java
@Value
public class AuthInitRequest {
    String initData;

    @JsonCreator
    public static AuthInitRequest of(@JsonProperty("initData") String initData) {
        return new AuthInitRequest(initData);
    }
}
```

**Сложный DTO из нескольких источников:** сервис сначала собирает все данные (БД, initData и т.д.), затем один вызов фабрики.

```java
// В сервисе: сначала баланс и флаг, потом один вызов с полными данными
UserDto user = UserDto.of(userMap, denyuzhki, sokrovishcha);
return new AuthInitResponse(user, token, firstVisit);
```

## Где лежат DTO

Пакет `com.its.domovenok.core.dto`: все классы ответов и запросов API следуют описанному подходу.
